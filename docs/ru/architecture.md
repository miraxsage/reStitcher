# Архитектура

Relix построен на фреймворке [Bubble Tea](https://github.com/charmbracelet/bubbletea), реализующем Elm-архитектуру для терминальных приложений. В основе лежит однонаправленный поток данных: **Model → Update → View**.

## Обзор Elm-архитектуры

Приложение следует трём принципам:

- **Model** -- единственный источник истины, хранящий всё состояние приложения
- **Update** -- чистая функция, принимающая текущую модель и сообщение, возвращающая обновлённую модель и команду
- **View** -- функция, преобразующая модель в строку для отображения в терминале

Все взаимодействия (нажатия клавиш, результаты API-вызовов, вывод git-команд) поступают в виде сообщений (`tea.Msg`), которые обрабатываются методом `Update`.

## Схема переходов экранов

```
                    ┌──────────────┐
                    │   Loading    │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │     Auth     │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐     ┌─────────────────┐
                    │     Home     │────▶│  History List    │
                    └──────┬───────┘     └────────┬────────┘
                           │                      │
                    ┌──────▼───────┐     ┌────────▼────────┐
                    │  MR Select   │     │  History Detail  │
                    └──────┬───────┘     └─────────────────┘
                           │
                    ┌──────▼───────┐
                    │  Env Select  │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   Version    │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │ Source Branch │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  Env Merge   │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  Root Merge  │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   Confirm    │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   Release    │
                    └──────────────┘
```

Модальные окна (командное меню, настройки, выбор проекта) накладываются поверх любого экрана и не являются отдельными состояниями в конечном автомате.

## Структура проекта

### Ядро

| Файл | Назначение |
|------|------------|
| `main.go` | Точка входа, парсинг флагов CLI, инициализация программы |
| `model.go` | Корневая модель, методы Init/Update/View, маршрутизация сообщений |
| `types.go` | Определения типов: экраны, окружения, MR, состояние релиза, сообщения |

### Экраны

| Файл | Назначение |
|------|------------|
| `auth_screen.go` | Аутентификация -- форма ввода учётных данных |
| `home_screen.go` | Главный экран -- меню действий |
| `mrs_screen.go` | Выбор MR -- список с фильтрацией и панелью деталей |
| `environment_screen.go` | Выбор окружения, версии, исходной ветки, мержа в окружение, root merge |
| `confirm_screen.go` | Подтверждение -- сводка параметров перед выполнением |
| `release_screen.go` | Выполнение релиза -- конечный автомат, терминальный вывод, мониторинг пайплайна |
| `history_screen.go` | История -- список и детали релизов |
| `settings_screen.go` | Настройки -- вкладки Release и Theme |

### Инфраструктура

| Файл | Назначение |
|------|------------|
| `gitlab.go` | GitLab API клиент -- проекты, MR, пайплайны |
| `git_executor.go` | Выполнение git-команд через PTY с виртуальным терминалом |
| `config.go` | Чтение/запись конфигурации и состояния релиза |
| `release_history.go` | Двухуровневое хранилище истории релизов |
| `keyring.go` | Безопасное хранение учётных данных через системный keyring |
| `theme.go` | Система тем -- разрешение цветов, ANSI-ремаппинг, фоновые стили |

### UI

| Файл | Назначение |
|------|------------|
| `styles.go` | Определения стилей Lipgloss |
| `utils.go` | Вспомогательные функции: перенос текста, парсинг версий, исключения файлов |
| `project_selector.go` | Модальный компонент выбора проекта |

## Ключевые паттерны

### Асинхронность на сообщениях

Все длительные операции (API-вызовы, git-команды) выполняются асинхронно через `tea.Cmd`. Команда возвращает функцию, которая выполняется в отдельной горутине, а результат приходит в виде `tea.Msg`:

- Загрузка MR, проектов, истории -- запуск через `tea.Cmd`, результат через типизированные сообщения (`fetchMRsMsg`, `fetchProjectsMsg` и т.д.)
- Состояние загрузки отслеживается булевыми флагами (`loadingMRs`, `loadingProjects`)
- Во время загрузки отображается спиннер

### Модальная система

Модальные окна накладываются поверх текущего экрана и управляются булевыми флагами:

- `showCommandMenu` -- командное меню
- `showProjectSelector` -- выбор проекта
- `showSettings` -- настройки

Функция `closeAllModals()` централизованно закрывает все модальные окна. Модальные окна перехватывают события клавиш до передачи обработчику экрана.

### Конечный автомат релиза

Процесс релиза реализован как конечный автомат с перечислением `ReleaseStep`:

```
Idle → GitFetch → CheckoutRoot → MergeBranches → CheckoutEnv →
CopyContent → Commit → PushBranches → WaitForMR → PushAndCreateMR →
WaitForRootPush → PushRootBranches → SwitchToRoot → Complete
```

Состояние сериализуется в `~/.relix/release.json` после каждого успешного шага. При сбое или прерывании процесс возобновляется с последней контрольной точки. Файл состояния удаляется только при успешном завершении или явной отмене пользователем.

### Git Executor

`GitExecutor` выполняет git-команды через псевдотерминал (PTY) с парсингом ANSI-последовательностей через библиотеку vt10x:

- Команды отправляются через каналы, вывод стримится обратно в UI через `tea.Msg`
- Виртуальный терминал эмулирует реальное терминальное окружение для корректного цветного вывода
- Размеры виртуального терминала синхронизируются с размерами viewport

### Двухуровневая история

Хранилище истории разделено на два уровня для оптимизации производительности:

1. **Индекс** (`index.json`) -- легковесный список с минимальным набором полей (версия, окружение, дата, статус) для быстрого отображения списка
2. **Детали** (`{timestamp}.json`) -- полные данные релиза, включая терминальный вывод, URL мерж-реквестов, SHA коммитов и конфигурацию

Детали загружаются по запросу, только при переходе к конкретной записи.
